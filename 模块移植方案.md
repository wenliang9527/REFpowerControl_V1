# REFpowerControl → REFpowerControl_V1 模块移植方案

---

## 一、项目概况

| 项目 | 源项目 (REFpowerControl) | 目标项目 (REFpowerControl_V1) |
|------|--------------------------|-------------------------------|
| MCU型号 | AT32F403A (Cortex-M4) | AT32F403AVGT7 (Cortex-M4) |
| 系统主频 | 240 MHz (HICK×PLL60) | 240 MHz (HICK×PLL60) |
| Flash | 1MB | 1MB |
| SRAM | 96KB | 96KB |
| 编译器 | ARMCC V5 | ARMCC V5.06 |
| 驱动库 | AT32F403A_407 StdPeriph | AT32F403A_407 StdPeriph (相同) |
| 项目状态 | 功能完整 | BSP框架已搭建，业务逻辑未移植 |

### 1.1 核心结论

- **芯片相同**：两个项目使用同一系列芯片 AT32F403A_407，驱动库完全一致，不存在芯片级不兼容问题
- **引脚重定义**：V1版本PCB硬件布局不同，GPIO引脚分配全面变更
- **新增外设**：V1新增 TMR1、DMA1_CH1/CH2、GPIOE端口
- **通信参数变更**：USART1从 19200/9E1 改为 115200/8N1 + DMA

## 五、移植实施步骤

### 阶段一：基础框架适配（P0）

#### 步骤1：确认V1基础模块无需修改

V1项目已有的基础模块（wk_system, wk_config, wk_debug, wk_gpio, wk_tmr, wk_usart, wk_dma, at32f403a_407_int）已按V1硬件配置完成，**无需修改**。

需确认以下内容：
- [x] wk_system：延时函数与源项目一致
- [x] wk_config：时钟配置、NVIC配置、引脚宏定义已更新
- [x] wk_debug：JTAG禁用逻辑一致
- [x] wk_gpio：所有引脚已按V1硬件初始化
- [x] wk_tmr：TMR1~4已初始化（TMR3/TMR4参数与源项目不同，需后续适配）
- [x] wk_usart：115200/8N1+DMA已配置
- [x] wk_dma：DMA1_CH1/CH2已初始化

#### 步骤2：调整定时器参数

源项目TMR3用于数码管扫描(0.5ms周期)，V1当前TMR3为1ms周期。需评估：

| 定时器 | 源用途 | 源周期 | V1周期 | 适配方案 |
|--------|--------|--------|--------|----------|
| TMR1 | - | - | 0.1ms | 预留给步进电机 |
| TMR2 | 制冷PWM | 1ms | 1ms | 无需修改 |
| TMR3 | 数码管+LED | 0.5ms | 1ms | **需修改预分频/周期**恢复0.5ms，或在数码管驱动中适配1ms |
| TMR4 | 传感器+按键 | 1ms | 1ms | 无需修改 |

**TMR3适配方案**：将V1的TMR3预分频从23改为59，周期从9999改为999，恢复0.5ms周期。
修改文件：`wk_tmr.c` 中的 `wk_tmr3_init()` 函数。

#### 步骤3：启用RTC/BPR模块

V1的 `at32f403a_407_conf.h` 中未启用 RTC 和 BPR 模块。需添加：

```c
#define RTC_MODULE_ENABLED
#define BPR_MODULE_ENABLED
```

并在Keil工程的 `firmware` 分组中添加：
- `at32f403a_407_rtc.c`
- `at32f403a_407_bpr.c`

#### 步骤5：移植 device_control 模块

| 项目 | 说明 |
|------|------|
| 源文件 | device_control.c / device_control.h |
| 移植方式 | 复制+引脚宏适配 |
| 适配工作 | 引脚宏已由V1的wk_config.h重新定义，无需修改源码 |
| 验证方法 | 控制各设备开关，用万用表/示波器验证GPIO输出 |

**关键适配点**：
- 源代码使用 `SW_REF_PIN`、`SW_FAN_1_PIN` 等宏，V1的 `at32f403a_407_wk_config.h` 已重新定义这些宏指向新引脚
- **无需修改 device_control.c 中的任何代码**，宏定义自动映射到新引脚
- 注意：V1新增了蜂鸣器 `BEEP_PIN`，需在 device_control 中扩展蜂鸣器控制接口

**操作**：
1. 将 `usercode/device_control.c` 和 `usercode/device_control.h` 复制到 `project/src/` 和 `project/inc/`
2. 在Keil工程 `user` 分组中添加 `device_control.c`

## 八、风险管理

### 8.1 技术风险识别与应对

| 风险编号 | 风险描述 | 影响等级 | 发生概率 | 应对措施 |
|----------|----------|----------|----------|----------|
| R01 | DS18B20引脚冲突：PA11/PA12在V1中分配给风扇控制 | 高 | 确定 | 确认V1温度传感器型号。如仍为DS18B20，使用PD10/PD11；如已更换，需重写驱动 |
| R02 | 数码管Charlieplexing扫描逻辑与V1引脚分布不兼容 | 高 | 中 | 数码管引脚从PB11~15+PA8变为PD9/PD8/PB15/PB14/PB13/PB12，跨端口操作可能影响扫描时序。需实测验证，必要时调整扫描间隔 |
| R03 | TMR3周期变更影响数码管刷新率 | 中 | 中 | 源项目0.5ms→V1项目1ms，刷新率从约450Hz降至约225Hz，可能出现闪烁。方案：恢复TMR3为0.5ms周期 |
| R04 | 触摸按键数量减少(9→4)导致功能缺失 | 中 | 确定 | 确认V1硬件是否仅需4个按键。如需更多功能键，需重新规划按键功能映射 |
| R05 | USART1通信参数变更(19200/9E1→115200/8N1)导致与上位机不兼容 | 高 | 中 | 确认上位机通信协议是否同步更新。如未更新，需将V1的USART1参数改回19200/9E1 |
| R06 | GPIO极性不一致：源项目设备控制为低电平有效，V1硬件可能不同 | 高 | 中 | 逐一确认V1硬件的GPIO有效电平，必要时修改device_control.c中的极性逻辑 |
| R07 | fluid_flow_interlock中直接读取GPIO端口状态，端口变更后可能读取错误 | 中 | 中 | 检查所有gpio_input_data_bit_read()调用，确认端口和引脚参数正确 |
| R08 | V1新增步进电机功能，与源项目无对应模块 | 低 | 确定 | 步进电机为新功能，不在本次移植范围内，预留TMR1和引脚即可 |
| R09 | DMA与USART1配合问题：源项目无DMA，V1新增DMA收发 | 中 | 低 | DMA收发由V1框架处理，业务模块仅需调用串口发送/接收接口。需确认接口兼容性 |
| R10 | V1的conf.h精简了模块启用，可能遗漏必要模块 | 低 | 中 | 移植rtc模块时需启用RTC/BPR/PWC，移植其他模块时按需启用 |

        ├── 3路传感器消抖(50ms)
